## Interpretation

There is no behavioral session dataset available in the current production database. That means the project cannot yet rank UX issues by observed drop-off, failed searches, selected results, repeated feedback, profile usage, saved-criteria reuse, or admin diagnostic usage.

This is a product signal by itself: either traffic is not reaching the persisted session flow, telemetry is pointed at a different database/environment, or users are not starting sessions.

## Static Workflow Evidence

- `public/index.html` exposes the public request flow, optional profile selection, cookie banner, legal links, and EU AI Act transparency notice.
- `public/test.html` exposes session creation, query submission, results, chat, profile selection, saved criteria, and an admin-flow link.
- `public/admin.html` is the admin diagnostic surface for prompts, models, settings, and agent communication flow.
- `src/sessions/` persists sessions, messages, search runs, search results, choices, and agent communication records.
- `src/profiles/` supports account profiles and saved search criteria.
- `prisma/schema.prisma` has the data needed for privacy-safe aggregate UX metrics once usage exists.

## Prioritized Recommendations

1. Add an aggregate-only UX metrics job or admin report.
   Evidence type: schema/static workflow.
   Reason: the system has rich tables but no repeatable report that counts funnel steps without exposing content. Track sessions, queries, result counts, zero-result runs, feedback messages, choices, profile usage, saved-criteria usage, and agent errors.

2. Make the empty-state path measurable.
   Evidence type: aggregate database result.
   Reason: current production data has zero persisted usage. Add a deployment-safe check that confirms the public landing/test flows create `Session`, `Message`, `SearchRun`, and `SearchResult` records in the expected database.

3. Clarify profile and saved-criteria onboarding in the test/public flow.
   Evidence type: static workflow.
   Reason: profile and saved-criteria features exist but depend on a saved JWT/admin-token style flow. Users may not understand when profile selection is available or why it is hidden.

4. Add zero-result and recovery messaging to the user-facing result state.
   Evidence type: SA-G1-T1 implementation plus static results UI.
   Reason: search recovery now exists in `SearchService`; the UI should distinguish "still searching with a broader query" from final "no usable results" when future behavior instrumentation confirms this is common.

5. Keep admin diagnostics protected but easier to reach after test sessions.
   Evidence type: static workflow.
   Reason: `test.html` links to `admin.html?sessionId=...#flow`, which is useful for debugging. Preserve JWT protection, but add aggregate diagnostic summaries so admins do not need to inspect raw agent messages first.

## Non-Recommendations

- Do not remove GDPR, ePrivacy/cookie, or EU AI Act transparency notices.
- Do not export raw user queries, voice transcripts, lead messages, profile names, or agent communication content for UX analysis.
- Do not weaken admin JWT protection to make diagnostics easier.
- Do not deploy changes without owner approval in the active session.

## Follow-Up Validation

After real or synthetic usage is generated, rerun aggregate metrics and compare:

- session count;
- sessions with at least one query message;
- search runs per session;
- zero-result rate;
- result count distribution;
- choice rate;
- feedback-after-results rate;
- profile and saved-criteria adoption;
- agent error counts by message type.
